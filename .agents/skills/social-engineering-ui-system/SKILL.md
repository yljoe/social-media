# Social Engineering UI System

用這個 skill 把所有 UI 工作收斂到同一套產品語言，不要每次重新決定風格。預設目標是「繁體中文企業情報控制台」，強調高可讀性、列表一目瞭然、穩定資訊架構、清楚預覽、可逐步擴充。

## Quick Start

1. 先讀 `references/current-frontend.md`。
2. 再讀 `references/design-rules.md`。
3. 如果任務涉及新頁面或大改版，再讀 `references/page-blueprints.md`。
4. 先判斷頁面類型，再套用對應版型：
   - dashboard
   - generation
   - management
   - logs or costs
5. 除非使用者明確要求全面重做，否則保留既有資訊架構，只改善可讀性、對齊、層級與互動。

## 核心原則

- 預設使用繁體中文介面文案。
- 預設使用繁中可用字體，不用只支援英數的字體當主字體。
- 優先做「可掃描」而不是「炫技」。
- 重要資訊用列表、表格、區塊分組呈現，不要塞進長段落。
- 生成類頁面一定要有預覽區，且與設定區同頁並列或上下相鄰。
- 管理類頁面一定要有清楚欄位順序、狀態標籤、最後更新時間與快捷操作。
- 每頁只保留一個主 CTA。
- 不要為了視覺效果犧牲操作清晰度。

## 工作流程

### 1. 先分類任務

把任務先歸到一種主頁型：

- `dashboard`
  - KPI、摘要卡、最近活動、總覽表格
- `generation`
  - 左側設定，中間或右側預覽，必要時加 stepper
- `management`
  - 列表、表格、搜尋、篩選、編輯器
- `logs-costs`
  - 記錄表格、狀態、時間、成本、錯誤細節

### 2. 決定骨架

此專案預設骨架如下：

- 左側固定導覽 `Sidebar`
- 上方頁面說明或全域狀態 `Header/Hero`
- 中央主工作區 `Workspace`
- 預覽、詳細資訊或編輯器放在主工作區的第二欄

如果是生成頁：

- 左欄放輸入參數與主操作
- 右欄放預覽、結果、洞察、JSON 或 render 狀態

如果是管理頁：

- 上方放標題、摘要、搜尋、篩選
- 下方放表格或 entity list

### 3. 套用資訊密度規則

- 列表列高要能快速掃描，不要過胖。
- 表格欄位順序盡量固定：
  - 名稱
  - 類型或狀態
  - Provider / 來源
  - 更新時間
  - 動作
- 每列至少有一個主識別資訊和一個次要識別資訊。
- 動作按鈕不可只在 hover 才能被察覺。
- 狀態一律用 badge，不要只靠文字顏色。

### 4. 套用字體與文案規則

- 主字體使用：
  - `"Noto Sans TC", "PingFang TC", "Microsoft JhengHei", sans-serif`
- 數字密集區、成本、代碼、ID 可使用：
  - `"Inter", "Noto Sans TC", "Segoe UI", sans-serif`
- 如需等寬資料可用：
  - `"JetBrains Mono", "SFMono-Regular", Consolas, monospace`
- 非必要不要用純英文展示字體當主 UI 字體。
- 文案預設繁體中文，專有名詞保留英文。
- 標題短而明確，按鈕用動作動詞，例如：
  - `建立專案`
  - `生成內容`
  - `套用 Provider`
  - `查看預覽`

### 5. 驗證

完成 UI 後至少檢查：

- 手機版不橫向溢出
- 表單有 label
- icon-only button 有 `aria-label`
- focus-visible 清楚可見
- 預覽區與設定區關係清楚
- 列表欄位能在 3 秒內掃描完重點

## 必守規範

### 導覽

- 側欄是第一級導覽，不要把主要模組藏進二級彈窗。
- 導覽名稱使用產品任務名稱，不用抽象詞。
- 當前頁要有清楚 active state。

### 生成頁

- 必須同頁看到設定與預覽。
- 支援 tab 切換預覽型態，例如郵件、落地頁、手機版。
- 重要洞察可放在次級卡片，但不能蓋過主預覽。

### 列表與表格

- 優先清楚，不追求花俏。
- 行內資訊最多兩層。
- 表頭用短標籤。
- 長文字要能 truncate 或 wrap，不讓版面爆掉。
- 空狀態要有下一步，不可只寫「沒有資料」。

### 視覺風格

- 風格定位是「企業情報控制台」，不是行銷頁。
- 用冷靜、可信、精準的配色。
- 允許有少量高光色做狀態或主 CTA，但不要整頁過亮。
- 陰影、圓角、玻璃感都要克制。

### 響應式

- 桌面版優先採左右雙欄。
- 平板版可改上下堆疊。
- 手機版導覽要可收合，表格要能轉卡片或可水平捲動但仍可操作。

## 禁止事項

- 不要把主要操作藏成 hover-only。
- 不要用與內容無關的炫目漸層。
- 不要混用多套視覺語言。
- 不要讓單頁同時出現太多主按鈕。
- 不要把生成結果藏到第二頁才看得到。
- 不要把管理列表做成難掃描的大卡片瀑布流。
